# 개발 파이프라인 가이드

## 개요

일관된 개발 프로세스를 위한 파이프라인 정의입니다.
모든 기능 개발은 이 파이프라인을 따라 진행합니다.

---

## 1. 파이프라인 단계

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          개발 파이프라인                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐  │
│  │ 요구사항 │──▶│  설계   │──▶│ 설계검수 │──▶│  구현   │──▶│ 구현검수 │  │
│  │  분석   │   │         │   │         │   │         │   │         │  │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘  │
│       │             │             │             │             │        │
│       ▼             ▼             ▼             ▼             ▼        │
│   requires/      spec/       체크리스트     코드/테스트    complete/   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

---

## 2. 단계별 상세

### 2.1 요구사항 분석 (PRD Analysis)

**목적**: 요구사항을 명확히 이해하고 문서화

**입력**: 사용자 요청, 기획 문서

**프로세스**:
1. 요구사항 명확화 질문
2. 기존 코드베이스 분석 (Serena 활용)
3. 영향도 분석
4. 체크리스트 기반 검증

**체크리스트**:
```markdown
## 요구사항 분석 체크리스트

### 기능 정의
- [ ] 핵심 기능이 명확히 정의되었는가?
- [ ] 사용자 스토리가 작성되었는가?
- [ ] 입력/출력이 정의되었는가?

### 범위 정의
- [ ] 포함 범위가 명확한가?
- [ ] 제외 범위가 명확한가?
- [ ] MVP와 추가 기능이 구분되었는가?

### 예외 케이스
- [ ] 에러 상황이 정의되었는가?
- [ ] 경계 조건이 파악되었는가?
- [ ] 보안 고려사항이 있는가?

### 기술 분석
- [ ] 기존 코드 영향도 분석 완료?
- [ ] 필요한 기술/라이브러리 파악?
- [ ] 예상 복잡도 평가 완료?

### 성공 기준
- [ ] 완료 기준이 명확한가?
- [ ] 테스트 기준이 정의되었는가?
- [ ] 성능 기준이 있는가?
```

**산출물**: `docs/requires/REQ-XXX-[기능명].md`

---

### 2.2 설계 (Design)

**목적**: 구현 가능한 수준의 설계 문서 작성

**입력**: 요구사항 문서

**프로세스**:
1. 아키텍처 설계
2. API 설계 (필요시)
3. 데이터 모델 설계 (필요시)
4. UI/UX 설계 (필요시)

**설계 문서 구조**:
```markdown
# [기능명] 설계 문서

## 1. 개요
- 목적
- 범위
- 관련 요구사항 (REQ-XXX 링크)

## 2. 아키텍처
- 컴포넌트 구조 (mermaid 다이어그램)
- 데이터 흐름
- 의존성

## 3. 인터페이스 정의
### 타입/인터페이스
```typescript
interface Example {
  id: string;
  // ...
}
```

### API (해당시)
| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | /api/xxx | ... |

## 4. 데이터 모델 (해당시)
- 스키마 정의
- 관계 정의

## 5. UI/UX (해당시)
- 화면 구조
- 사용자 플로우
- 컴포넌트 목록

## 6. 에러 핸들링
- 예상 에러 목록
- 처리 방법

## 7. 테스트 계획
- 테스트 범위
- 주요 테스트 케이스
```

**예시 코드 규칙**:
```markdown
## 예시 코드 작성 규칙

✅ DO:
- 인터페이스/타입 정의만 포함
- 핵심 로직의 의사 코드
- 시그니처만 정의

❌ DON'T:
- 전체 구현 코드
- 보일러플레이트 코드
- 스타일링 코드
```

**산출물**: `docs/spec/[분류]/[기능명].md`

---

### 2.3 설계 검수 (Design Review)

**목적**: 설계의 완전성과 구현 가능성 검증

**체크리스트**:
```markdown
## 설계 검수 체크리스트

### 완전성
- [ ] 모든 요구사항이 설계에 반영되었는가?
- [ ] 인터페이스가 완전히 정의되었는가?
- [ ] 에러 케이스가 고려되었는가?

### 일관성
- [ ] 기존 아키텍처와 일관성이 있는가?
- [ ] 네이밍 컨벤션을 따르는가?
- [ ] 코드 스타일과 맞는가?

### 구현 가능성
- [ ] 기술적으로 구현 가능한가?
- [ ] 성능 요구사항을 충족할 수 있는가?
- [ ] 리소스(시간/복잡도)가 적절한가?

### 테스트 가능성
- [ ] 테스트 가능한 구조인가?
- [ ] 모킹이 필요한 부분이 명확한가?

### 문서화
- [ ] 설계 의도가 명확히 설명되었는가?
- [ ] 다이어그램이 이해하기 쉬운가?
```

**통과 기준**: 모든 체크리스트 항목 충족

---

### 2.4 구현 (Implementation)

**목적**: 설계를 바탕으로 코드 작성

**입력**: 설계 문서

**프로세스**:
1. 설계 문서 확인
2. 기존 코드 분석 (Serena)
3. 코드 작성
4. 테스트 작성
5. 태스크 문서 업데이트

**구현 규칙**:
```markdown
## 구현 규칙

### 코드 품질
- TypeScript strict 모드
- 명시적 타입 선언
- 에러 핸들링 필수
- 주석은 "왜"를 설명

### 테스트
- 단위 테스트 필수
- 엣지 케이스 테스트
- 에러 케이스 테스트

### 커밋
- 작은 단위로 커밋
- 의미있는 커밋 메시지
- 관련 이슈 참조
```

**진행 상태 문서**: `docs/tasks/TASK-XXX-[기능명].md`

---

### 2.5 구현 검수 (Implementation Review)

**목적**: 구현의 품질과 요구사항 충족 검증

**체크리스트**:
```markdown
## 구현 검수 체크리스트

### 요구사항 충족
- [ ] 모든 기능 요구사항이 구현되었는가?
- [ ] 예외 케이스가 처리되었는가?
- [ ] 성능 기준을 충족하는가?

### 코드 품질
- [ ] 타입 안전성이 확보되었는가?
- [ ] 에러 핸들링이 적절한가?
- [ ] 코드 중복이 없는가?
- [ ] 네이밍이 명확한가?

### 테스트
- [ ] 단위 테스트가 작성되었는가?
- [ ] 테스트가 모두 통과하는가?
- [ ] 커버리지가 충분한가?

### 설계 준수
- [ ] 설계 문서를 따랐는가?
- [ ] 인터페이스가 일치하는가?
- [ ] 불필요한 변경이 없는가?

### 문서화
- [ ] 코드 주석이 적절한가?
- [ ] API 문서가 업데이트되었는가?
- [ ] 변경 사항이 기록되었는가?
```

**통과 후 산출물**: `docs/complete/DONE-XXX-[기능명].md`

## 4. 에이전트 역할 매핑

| 단계 | 주 담당 에이전트 | 보조 에이전트 |
|------|-----------------|--------------|
| 요구사항 분석 | PM | Explorer |
| 설계 | Architect | PM |
| 설계 검수 | PM + QA | - |
| 구현 | Developer | Explorer |
| 구현 검수 | QA | PM |
| 문서화 | Documenter | - |

## 5. 파이프라인 적용 예시

### 기능 요청: "로그인 기능"

```markdown
## 1. 요구사항 분석
사용자: "로그인 기능 구현해줘"

PM 에이전트:
- 체크리스트 기반 질문
  - "이메일/비밀번호 로그인인가요?"
  - "소셜 로그인이 필요한가요?"
  - "자동 로그인 기능이 필요한가요?"
- 기존 코드 분석 (Serena)
- docs/requires/REQ-001-login.md 생성

## 2. 설계
Architect 에이전트:
- 아키텍처 설계 (컴포넌트 구조)
- API 설계 (인증 엔드포인트)
- UI 설계 (로그인 폼)
- docs/spec/ 문서 생성

## 3. 설계 검수
PM + QA:
- 체크리스트 기반 검토
- 피드백 및 수정

## 4. 구현
Developer 에이전트:
- 설계 문서 기반 구현
- 테스트 코드 작성
- docs/tasks/TASK-001-login.md 업데이트

## 5. 구현 검수
QA 에이전트:
- 체크리스트 기반 검수
- 테스트 실행
- 피드백

## 6. 완료
Documenter:
- docs/complete/DONE-001-login.md 생성
- 관련 문서 업데이트
```
